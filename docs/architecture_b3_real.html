3 · Padrão A vs Padrão A+B (decisão para Ops B3)

Padrão A — Só instrumentação no código

Quando usar: Ops B3 não quer mais um agent no host do Control-M Agent.

  • Python emite tudo direto pro endpoint público Datadog (HTTPS).
  • ✅ APM funciona (tráfego direto)
  • ✅ Métricas funcionam (StatsD HTTP backend)
  • ✅ OpenLineage funciona (DatadogTransport)
  • ⚠️ Logs ficam só no sysout do Control-M — sem indexação Datadog
  • ⚠️ DBM precisa rodar em outro host (não há Agent local)
  • Latência maior (HTTPS pra cada flush vs UDP local)

Padrão A+B — Datadog Agent no host (recomendado)

Quando usar: POC séria, demo B3 com tudo visível.

  • Mesmo Python instrumentado, mas envia para Agent local.
  • ✅ APM via socket local (mais rápido)
  • ✅ DogStatsD UDP local (perda zero em picos)
  • ✅ Logs do sysout indexados automaticamente (file tail)
  • ✅ DBM pode rodar no mesmo Agent (Oracle/SQL Server)
  • ✅ OpenLineage proxy (com buffer/retry)
  • Requer aprovação Ops para instalar Datadog Agent

5 · Riscos e dependências externas

RiscoProbabilidadeMitigação
Org Datadog da B3 sem SKU Data Jobs Monitoring (Preview) Alta Pedir CSM enable. Caso não rolar, lineage view fica vazia mas APM/DBM/Logs/Dashboard customizado seguem funcionando.
Ops não aprovar Datadog Agent no host do Control-M Agent Média Padrão A puro. Demo perde Logs indexados, mas mantém APM/Métricas/OL/DBM.
Control-M abaixo da versão 9.0.20 Baixa (confirmar com Lairton) Job:Command sempre funciona. Só perde Automation API REST — JSONs viram XML legado.
OpenLineage Python 1.37+ DatadogTransport bug em produção Baixa Fallback HTTP transport (mesmo código) — emite direto pro /openlineage/api/v1/lineage do Agent.
Grants DBM em Oracle XE com bug OP_FLAGS (que vimos) N/A em produção Oracle Enterprise (B3) não tem o bug do XE 21c. query_samples.enabled: true funciona normalmente.
Resumo executivo: esta arquitetura usa apenas componentes já validados em produção em Control-M + Datadog (não depende de plug-ins custom da BMC, não depende de Application Integrator). As únicas novidades para o time Ops da B3 são (1) editar o campo Command do job para incluir ddtrace-run e (2) opcionalmente instalar o Datadog Agent no host do Control-M Agent. O caminho mais arriscado é o lineage stand-alone (Preview / CSM enable) — todos os outros sinais (APM, DBM, métricas, logs, dashboard customizado) são GA.
